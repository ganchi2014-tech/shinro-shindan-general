# データ検証台帳（v2）

v2データ（work/data_v2.json）の実データ検証の記録。**2027年3月の年次更新時はこのフォルダを作業台帳として使う。**

## 構成
- `PROMPT_TEMPLATE.md` — 調査エージェントに渡すプロンプトの雛形
- `<uni_id>.md` — 大学ごとの照合レポート（調査日・出典・変更差分）
- `diff_log.jsonl` — 全変更の機械可読ログ（追記式）

## 運用フロー（1バッチ = 20〜30校）
1. `work/dept_manifest.json` から対象校の学部一覧を切り出す（バッチ生成コマンドは実装計画 Task 6〜10 参照）
2. PROMPT_TEMPLATE.md に大学名・学部一覧を埋めて調査エージェントを並列起動（1エージェント3〜5校）
3. エージェント出力を `work/research/<uni_id>.json` に保存
4. `cd work && node merge_research.js research/<uni_id>.json ...` でマージ（部分失敗時は何も保存されない。全ファイル修正後に再実行）
5. `node check_v2.js` で検収 → コミット

## 情報源の優先順（spec準拠）
1. 河合塾Kei-Net（偏差値の正）
2. 大学公式サイト（入試方式・倍率・ゼミ・資格）
3. パスナビ等の進学情報サイト（補完）

## 原則
- 取れなかった値は無理に埋めない（unconfirmed運用）
- 全値に source_url を付ける。付けられない値は書かない
- ゼミは学部あたり最大3件・看板を先頭に。深掘り調査はしない
- `unconfirmed` は**学部オブジェクト直下**のフィールド（hensachiの中など、ネストしたオブジェクトの中に置かない）
- `qualifications` / `employment_fields` は調査しなかった・確認できなかった場合は**キーごと省略**する。空配列 `[]` を送ると「0件が確定した」という意味になり、既存値を消してしまうため、確信がある場合以外は `[]` を送らない
- `entry_methods` は判明した方式のみ記載する。`available` を確認できなかった方式は**エントリごと省略**する（noteだけ送りたい場合も `available` は必須）
