# 進路診断v2 フェーズ3：スマホ特化UI 設計

- 日付: 2026-07-15
- 前提: [[shinro-v2-phase2-engine]]（engine_v2.js `diagnose` 完成）／フェーズ1データ基盤（data_v2.json・650学部）
- 位置づけ: 案Aハイブリッド3フェーズの最終フェーズ（①データ基盤→②診断エンジン→**③スマホ特化UI**）

## ゴール

engine_v2 の `diagnose()` を組み込んだ本番UIを、v1の実績あるスマホSPA構成を継承して構築する。新価値は **RIASEC適性入力** と **3ゾーン（挑戦/実力/安全）結果表示**。

## 方針（採用＝案A）

v1で完成している資産（PWA・ダークテーマ・画面遷移ウィザード・チップ入力・お気に入り・詳細画面）を継承し、次の3点に集中する:

1. RIASEC適性入力（12問）の追加
2. 診断ロジックを engine_v2 に載せ替え（v1のインライン診断ロジックを撤去）
3. 結果を3ゾーン表示に再設計

不採用: ゼロから全面再構築（工数大・リスク高）／v1に3ゾーンだけ後付け（適性という新機軸が活きない）。

## アーキテクチャ

- **単一HTML・PWA・オフライン・バニラJS**（v1構成を維持）。フレームワーク不使用。
- **v1とは別ファイルで新規配布**: 出力は `進路診断_v2.html`。v1（`進路診断_一般版.html`）は2027年3月まで併存公開のため無変更。
- `engine_v2.js` と `data_v2.json` を**両方インライン**して単一ファイル化。
- 想定サイズ: data_v2 約1.6MB＋engine＋UI ≒ **約2.3MB**。オフライン配布・QR配布に問題なし。
- テーマ・CSS変数・PWAメタ（apple-mobile-web-app等）はv1から継承。

### ビルド（build_v2.js 新設）

v1 `build.js` を踏襲。テンプレート `app_v2.html` に対し:

1. `<script type="application/json" id="inlineData">` に `data_v2.json` を流し込む。
2. `<script id="engineSrc">` プレースホルダに `engine_v2.js` のソースを流し込む（require/module.exportsをブラウザ用にshim、またはIIFEでグローバル公開）。
3. 検証: メインJSの `new Function` 構文チェック＋`JSON.parse(data)`＋engineの必須関数（`diagnose`等）存在チェック。
4. 出力: `../進路診断_v2.html`、サイズをログ。

engine_v2.js 本体は無変更（フェーズ2のシェバン削除済み＝ブラウザ評価可）。ブラウザ用取り込みはハーネス同様、UI側で `module` をshimして評価する（engine本体を汚さない）。

## 画面フロー

```
ホーム
 └ 診断スタート
     ① 偏差値・模試ソース（v1流用：クイックボタン＋「わからない」）
     ② エリア（ハードフィルタ：地域チップ＋「通学可のみ」）
     ③ 興味分野14 / 希望就職系統12 / 使える入試方式（v1流用チップ）
     ④ RIASEC適性 12問（0〜3）★新規　←「スキップ可」
     ⑤ loading → 結果
結果（偏差値ありなら3ゾーン／偏差値未入力ならfit順1リスト）
 └ 学部カード タップ → 詳細画面（v1流用）
 └ お気に入り（v1流用）
```

- ①〜③はv1入力画面を流用し、値の受け渡しのみ engine_v2 の `userProfile` 契約に合わせて差し替え。
- ④RIASECは**スキップ可**。`riasecAnswers=[]`→`answered=false`。さらに興味も未選択なら `meta.lowSignal=true`（精度向上の導線を出す）。なお1本リスト表示になるのは**偏差値未入力のとき**であって、スキップそのものが直接ungroupedにするわけではない（偏差値を入れていればスキップでも3ゾーン＝興味・就職主導のfitで分類）。時間のない生徒はスキップ、意欲的な生徒は12問で適性精度向上、の二段構え。
- **重要（engineの実挙動）**: `answered` は「配列が空でないか」で決まる。よって**全問中立(1)のまま「12問で診断」した場合も `answered=true`＝3ゾーン診断**になり、lowSignalにはならない。スキップは必ず**空配列 `[]` を送る専用ボタン**で行う（下記UI節）。

### userProfile 契約（engine_v2 入力）

`diagnose(userProfile, dataV2, tagsDef, riasecMap, opts?)` に渡す `userProfile`:

| フィールド | UIの供給元 |
|---|---|
| `hensachi` (number|null) | ① 偏差値入力（「わからない」でnull） |
| `examSource` (string) | ① 模試ソース選択（kawai/shinken/sundai/toshin） |
| `regions` (string[]) | ② 地域チップ選択 |
| `commuteOnly` (bool) | ② 「通学可のみ」チェック |
| `interests` (string[]) | ③ 興味チップ |
| `careers` (string[]) | ③ 就職系統チップ |
| `examMethods` (string[]) | ③ 入試方式チェック |
| `riasecAnswers` (number[12]) | ④ RIASEC回答。回答時は12要素の配列。**スキップ時は空配列 `[]`**（→ `answered=false`。興味も未選択なら `lowSignal`） |
| `budgetMax` (number|null) | 予算（v1にあれば流用、なければnull固定・将来拡張） |

`riasecMap` は `interest_riasec_map.json` の `map` 配下を渡す。`opts.topN` 既定12。

## RIASEC入力UI（新規・肝）

- 12問を **1問1カード・縦スクロール**で提示（v1のステップ内スクロールに合わせる）。
- 各問は **0〜3の4段ボタン**（「あてはまらない / ややあてはまる / あてはまる / とてもあてはまる」）。スライダーよりタップ数が少なく誤操作が少ない。
- 進捗は「◯/12」の小表示。未回答の問は内部的に中立(1)で埋めて `riasecAnswers` を12要素で構築する（部分回答でも成立）。
- ボタンは2つ:
  - **「この回答で診断」** → 12要素配列を送る（`answered=true`＝3ゾーン）。誤って全問未回答のまま押した場合はホランド表示が無意味になるため、**最低1問の回答を促す軽いガード**を入れる。
  - **「スキップ（偏差値順で見る）」** → `riasecAnswers=[]` を送る（`answered=false`）。興味未選択なら lowSignal→ungrouped、興味選択済みなら興味主導の3ゾーン。
- 設問文は `riasec_def.json` の `items` から描画（ハーネスと同じ配線）。

## 結果画面（3ゾーン・目玉）

- 上部 **サマリー帯**: 換算偏差値・ホランドコード（例「RIA」＋一言解説）・該当件数。
- 本体 **3ゾーンを縦セクション**:
  - 🚀 挑戦（gap +2.5〜+10）
  - 🎯 実力相応（±2.5）← 既定で最初に目に入る位置
  - 🛡️ 安全（−2.5〜−10）
- **学部カード**: 大学名・学部学科／偏差値帯／倍率／**適性fitスコア**（バッジ）／**適性理由チップ**（🧭適性○ 📚興味一致 💼就職 🚃通学圏）。タップで詳細へ。
- **表示分岐は「偏差値の有無」で決める（lowSignalではない）**:
  - **偏差値あり** → `zones`（挑戦/実力/安全）を表示。engineは偏差値ありなら各学部をゾーン分類する。
  - **偏差値null** → engineは全学部を `ungrouped`（**fitスコア順**）に入れる（ゾーン基準が作れないため）。この場合は3ゾーンをやめ**1本リスト**で表示。
  - 判定条件: `meta.adjustedHensachi === null`（＝偏差値未入力）で1本リスト、そうでなければ3ゾーン。
- **`lowSignal` はメッセージ専用**: `meta.lowSignal===true`（RIASEC未回答かつ興味未選択）のとき、結果上部に「適性診断や興味選択をするとより精度が上がります」導線を出す。表示レイアウトの分岐には使わない。
- **混在ケース**: 偏差値ありでも学部側に偏差値欠損がある学部はengineが `ungrouped` に入れる。この場合は3ゾーンの下に「偏差値データなし」小セクションとして `ungrouped` を追記表示する。
- 各ゾーンは `opts.topN`（既定12）で上限。件数が多い場合の「もっと見る」は将来拡張（YAGNI、初版は topN 表示）。

## エラー・境界処理

- 偏差値null: 上記「表示分岐」に従い **fitスコア順の1本リスト（ungrouped）** を表示。加えて `lowSignal` なら精度向上の導線を出す。UI側は分岐と描画のみでソート等はengineに委ねる。
- 全ゾーン0件かつungroupedも0件（エリア極小＋偏差値帯外など）: 「条件に合う学部が見つかりませんでした。エリアや偏差値を見直してください」＋除外内訳（byRegion等）を控えめに表示。
- data_v2 ロード失敗（インラインのため通常起きないが）: エラーメッセージ表示。

## テスト方針

- engine_v2 のロジックはフェーズ2でNode TDD検証済み。**UI層では触らない**。
- UI結合は engine_harness.html の配線を本番UIへ昇格させる形。**ブラウザ実機で目視確認**:
  1. 各入力→`userProfile` 生成が契約通りか
  2. 3ゾーン描画（挑戦/実力/安全のカード・fit・理由チップ）
  3. スキップ時の ungrouped 1リスト表示
  4. 学部カード→詳細遷移
  5. 0件時・null偏差値時の分岐
- build_v2.js に **JS構文チェック＋JSON妥当性＋engine必須関数存在チェック**を組み込み、ビルド時に破損検出（v1 build.js踏襲）。
