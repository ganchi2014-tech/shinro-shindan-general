# 診断エンジンv2（フェーズ2）設計

**日付**: 2026-07-15
**位置づけ**: 進路診断一般版v2の案Aハイブリッド3フェーズのうち②。①学部単位データ基盤（フェーズ1、`data_v2.json` verified 642/650）は完成済み。本フェーズは診断ロジックのみを純粋モジュールとして作る。③スマホ特化UI全面再設計はフェーズ3。

## ゴール

学部単位データ（`work/data_v2.json`：大学123＋学部650）を使い、**ハードフィルタ（地域）＋RIASEC適性＋挑戦/実力/安全の3段**で学部をランキングする診断エンジンを、テスト付き純粋モジュールとして実装する。v1の弱点（大学単位・全要素を1スコアに混ぜて「なぜ上位か」が不透明）を、**学部単位・2軸分離（適性fit と 到達難易度zone を独立）**で解消する。

## 非ゴール（YAGNI / フェーズ3送り）

- 本番UI・画面遷移・スマホ最適化（フェーズ3）
- v1アプリ（`進路診断_一般版.html` / `work/app.html`）の改変（**engineは読み取り専用**、v1・data_v2本体は無変更）
- interest_tags タクソノミーの再定義（暫定14のまま。RIASEC重み表で適性に接続する）
- ゼミ・就職の追加調査（既存data_v2の範囲で完結）
- unconfirmed 8学部の救済（偏差値なしのため比較対象外＝除外運用）

## アーキテクチャ

**成果物（すべて `work/` 配下、依存パッケージなし・Node素のassert）**

| ファイル | 役割 |
|---|---|
| `engine_v2.js` | 純粋関数モジュール。公開API `diagnose()` ＋ 単体テスト可能な小関数を分割export（`computeRiasecProfile` / `hardFilter` / `classifyZone` / `computeFit`） |

**設計原則**：ロジックとUIを完全分離。`engine_v2.js` はDOMもfetchも持たず、引数で受け取ったデータだけで計算する純粋関数。フェーズ3のUIはこのモジュールを import して使う。

## 公開API

```js
diagnose(userProfile, dataV2, tagsDef, riasecMap) → result
```

### 入力 userProfile

```js
{
  hensachi: 55 | null,            // 河合換算前の模試偏差値。nullなら偏差値未入力
  examSource: 'kawai' | 'shinken' | 'susumu' | ...,  // 模試種別（河合換算オフセット用）
  regions: ['shiga', 'kyoto', 'osaka'],   // 検討するエリア（data_v2の region ID）
  commuteOnly: false,            // trueなら通学可(commute_possible)の大学のみ
  riasecAnswers: [3,2,0,...12],  // 12設問の回答(0-3)。空配列/undefinedなら未回答
  interests: ['keizai', 'joho'], // 選択した興味タグ（0個可）
  careers: ['it', 'kinyu'],      // 希望就職系統（employment 12系統、0個可）
  examMethods: ['ippan','kyotsu'], // 使える入試方式（sogo/koubo/ippan/kyotsu）
  budgetMax: null                // 年間学費上限（円）。nullなら無制限
}
```

### 出力 result

```js
{
  zones: {
    challenge: [item, ...],   // gap +2.5〜+10、fit降順、既定上位12件
    match:     [item, ...],   // gap −2.5〜+2.5
    safe:      [item, ...],   // gap −10〜−2.5
  },
  ungrouped: [item, ...],     // hensachi=null時のみ（zonesは空、全件fit降順）
  excluded: { byRegion: n, unconfirmed: n, outOfRange: n },  // 足切り件数（理由表示用）
  meta: {
    adjustedHensachi: 53 | null,
    riasecProfile: { R, I, A, S, E, C },  // 各0.0-1.0
    hollandCode: 'SIC',        // 上位3型
    lowSignal: false           // RIASEC未回答かつ興味未選択
  }
}

// item
{
  dept_id, uni_id, uni_name, dept_name, campus,
  hensachi: { min, max, mid },
  bairitsu,                    // data_v2の bairitsu をそのまま
  zone: 'challenge' | 'match' | 'safe',
  fitScore,                    // 0-100
  reasons: [ '🧭 適性◎ …', ... ],  // 最大5件
  breakdown: { appt, emp, soft, riasecFit }
}
```

## パイプライン（段階分離）

エンジンは以下を順に実行する。各段は独立関数。

### 1. ハードフィルタ `hardFilter`
- `userProfile.regions` に含まれない大学の学部を除外（`excluded.byRegion` に計上）
- `commuteOnly` が true の場合、大学の `commute_possible !== true` を除外
- `data_status === 'unconfirmed'` の学部を除外（偏差値なしで比較不能。`excluded.unconfirmed`）
- **地域のみがハードフィルタ**。入試方式・学費・興味は除外せずスコアの加減点（ソフト）で扱う

### 2. ゾーン分類 `classifyZone`
- `mid = (hensachi.min + hensachi.max) / 2`、`gap = mid − adjustedHensachi`
- `adjustedHensachi = hensachi + EXAM_SOURCE_OFFSET[examSource]`（河合換算。v1の `EXAM_SOURCE_OFFSET` を踏襲）
- 挑戦: `+2.5 ≤ gap ≤ +10` / 実力: `−2.5 ≤ gap < +2.5`（境界は実力側に寄せる） / 安全: `−10 ≤ gap < −2.5`
- `|gap| > 10` は現実的射程外として `excluded.outOfRange` に計上しゾーンに入れない
- `hensachi === null`（偏差値未入力）時はゾーン分けせず、全通過学部を `ungrouped` に fit 降順で入れる

### 3. 適合スコア `computeFit`（偏差値は入れない。0-100）

**① 適性ブロック（最大55）**
- `riasecFit`（0-1）= その学部が必要とするRIASEC型を、ユーザーがどれだけ満たすかの加重平均。学部ベクトル `dv` = その学部の `interest_tags` の重み（`interest_riasec_map.json`）を型ごとに平均。ユーザープロフィール `uv`（各0-1）との**重み付き内積を学部側重み和で正規化**：`riasecFit = Σ(uv[t]·dv[t]) / Σ(dv[t])`（`Σdv[t] = 0` すなわち unclassified のみなら 0）。→ **×40**
- `interestMatch`（0-1）= 学部 `interest_tags` ∩ ユーザー `interests` の重なり率（ユーザー選択0個なら0）→ **×15**

**② 就職ブロック（最大25）**
- 学部 `employment_fields` ∩ ユーザー `careers` の重なり：1件 +8、上限25

**③ ソフト条件ブロック（最大20・減点あり）**
- 入試方式：`entry_methods` で available かつ ユーザー `examMethods` に含まれる方式あり = **+6** ／ 学部の実施方式が一つも噛み合わない = **−10**
- 学費：`budgetMax` 指定かつ 大学 `tuition_yen_per_year` 超過 = **−8**（除外しない）
- ゼミ：`zemi.length > 0` = **+4**
- 通学利便：大学が滋賀県内 = **+8** ／ `commute_possible === true` = **+5**（排他、大きい方）

`fitScore = clamp(適性 + 就職 + ソフト, 0, 100)`

**reasons**：寄与の大きい項から最大5件を生成（例「🧭 適性◎ 研究×情報 (SIC)」「💼 IT・通信への就職に強い」「🎓 看板ゼミあり」「🏞️ 通学圏」）。

**lowSignal**：`riasecAnswers` 未回答 かつ `interests` 未選択なら `meta.lowSignal = true`（就職・ソフトのみで暫定ランク。UI側で注意表示できるように）。

### 4. 整形
- ゾーンごとに fit 降順ソート、既定で各ゾーン上位12件（引数で可変にしてよい）
- 各 item に reasons・breakdown を付与

## RIASEC質問票（`riasec_def.json`）

各タイプ2問、計12問。4段階回答（3=とても当てはまる / 2=やや / 1=あまり / 0=当てはまらない）。

| type | 設問 |
|---|---|
| R 現実的 | 機械・道具を分解/修理/組立てするのが好き ／ 体を動かす作業や屋外・現場の活動にやりがい |
| I 研究的 | 物事の仕組みや「なぜ」を突き止めるのが好き ／ データや実験で仮説を確かめる作業が苦にならない |
| A 芸術的 | 絵・音楽・文章・映像など何かを表現するのが好き ／ 決まったやり方より自由な発想で作る方が向く |
| S 社会的 | 人に教えたり相談に乗って助けるのが好き ／ 誰かの役に立てたと感じる仕事にやりがい |
| E 企業的 | 人をまとめ企画を提案してリードするのが好き ／ 目標を決め競争しながら成果を出すのにワクワク |
| C 慣習的 | 計画を立て正確・丁寧に進めるのが得意 ／ 数字やルールに沿ってこつこつ整理・管理が好き |

**算出（`computeRiasecProfile`）**：`profile[type] = (設問a + 設問b) / 6`（各0.0-1.0）。上位3型を連結して `hollandCode`（例 'SIC'）。未回答なら全型0＋lowSignal。

## テスト方針

各ステップTDD（テスト先行）。素のNode assert（`node test_engine_v2.js`）。

- `computeRiasecProfile`：12回答→各型0-1・ホランドコード上位3／未回答→全0＋lowSignal
- `hardFilter`：エリア外除外・unconfirmed除外・commuteOnlyの件数
- `classifyZone`：gap閾値（挑戦+2.5〜+10／実力±2.5／安全−2.5〜−10／|gap|>10は outOfRange）・偏差値null→ungrouped
- `computeFit`：RIASEC内積・興味重なり・就職重なり・ソフト加減点・0-100クランプ・reasons生成
- `diagnose` 統合：小さな合成フィクスチャ（数学部）で ゾーン所属・並び順・reasons・excluded集計を検証
- 実data_v2スモーク：サンプルプロフィールで実行し、ゾーンが空でない／エリア制限時 `excluded.byRegion > 0`／クラッシュ無しを確認

## 引き継ぎ（フェーズ3へ）

- フェーズ3のUIは `engine_v2.js` の `diagnose()` を import して結果を描画する。入出力契約は本specの通り。
- `EXAM_SOURCE_OFFSET` はv1の値を engine_v2 に移植（模試種別→河合換算）。
- 重み（40/15/25/20、gap閾値、RIASEC重み表）はテスト・目視で微調整余地あり。定数は engine_v2 冒頭にまとめ、調整しやすくする。
