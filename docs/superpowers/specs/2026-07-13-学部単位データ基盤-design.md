# 進路診断アプリ一般版 v2 — フェーズ1: 学部単位データ基盤 設計書

**日付**: 2026-07-13 / **対象**: `進路診断_一般版\work\` のデータ層のみ（アプリ本体・UIは触らない）

## 背景と全体計画

v1（2026-07-10完成、123校・単一HTML・GitHub Pages公開中）を、v2として全面リニューアルする。全体は3フェーズ:

1. **フェーズ1（本書）**: 学部単位データ基盤 — 大学単位→学部単位の2層データ＋実データ検証＋看板ゼミ
2. フェーズ2: 診断エンジンv2 — ハードフィルタ→適合度ランキングの2段階、適性診断の深化（RIASEC的プロフィール）、挑戦/実力/安全の3段提示
3. フェーズ3: UI全面再設計 — スマホファースト、結果画面のビジュアル強化、トーン刷新、単一HTMLビルド維持

アプリ本体は新規書き直し、データ資産とビルドパイプライン（work/）は拡張流用。v1は完成まで併存公開（有効期限2027年3月末）。

## フェーズ1の目的

- 大学単位（123校）のデータを、**学部単位（約650学部）を主役とする2層構造**に再編する
- 偏差値・倍率・入試方式をWeb調査で実データ検証し、**出典URL＋検証日**を全値に記録する
- **全学部に看板ゼミ・研究室を各1件**付属させる（特定できない学部はフォールバック）
- 診断エンジン・UIには触らない

## 現状（出発点）

- ippanカテゴリに既に**650学部分の簡易データ**（学部名・キャンパス・hensachi_min/max・qualifications）が大学にぶら下がる形で存在 → これを一級レコードへ昇格させる

## 新スキーマ（2層構造）

### 大学レコード（123校・共通情報を維持）

現行baseの内容をほぼ継承: 所在地・地域・国公私区分・アクセス（滋賀基点）・生活費・奨学金・キャンパス周辺・就職実績・キャリアセンター・オープンキャンパス・留学・公式リンク等。

### 学部レコード（約650件・新設の主役）

```jsonc
{
  "dept_id": "ritsumeikan-keiei",        // 大学ID + 学部スラッグ
  "uni_id": "ritsumeikan",
  "name": "経営学部",
  "campus": "大阪いばらき",               // 学部ごとに正確に
  "hensachi": {
    "min": 57.5, "max": 60,
    "basis": "kawai",                     // 河合基準に統一
    "source_url": "...", "verified_date": "2026-07-xx"
  },
  "bairitsu": {                           // 一般入試の直近2年分
    "ippan": { "2025": 3.2, "2026": 2.9 },
    "source_url": "...", "verified_date": "..."
  },
  "entry_methods": [                      // 学部単位の入試方式
    { "type": "sogo" | "koubo" | "ippan" | "kyotsu", "available": true, "note": "..." }
  ],
  "qualifications": ["公認会計士", "税理士", "中高教員"],
  "interest_tags": ["経済経営"],          // 適性マッチング用・暫定タクソノミー
  "employment_fields": ["金融", "メーカー"], // 主な就職系統 2〜4個
  "zemi": [ { "name": "○○ゼミ", "theme": "テーマ1行", "source_url": "...", "verified_date": "..." } ],
  // 配列（最低1件を目標、調査中に負担なく複数見つかれば最大3件まで記載）
  // 空配列可（→大学単位 zemi_features にフォールバック）
  "data_status": "verified" | "estimated" | "unconfirmed"
}
```

### 設計上の注意

- **interest_tags のタクソノミーは暫定**（現行14分野ベース）。フェーズ2の適性診断設計で確定後、一括リタグする。タグはIDで持ち、タグ定義表を別途1箇所にまとめてリタグを機械的にできるようにする
- 大学レコードの `zemi_features`（1行テキスト）は削除せず残す — 学部zemiが null の場合の表示フォールバック
- v1のデータカテゴリ構造（sogo/koubo/ippan別）は廃止し、入試方式は学部レコードの `entry_methods` に一本化

## 実データ検証の方法

- **情報源の優先順**:
  1. 河合塾Kei-Net — 偏差値の正とする
  2. 大学公式サイト — 入試方式・倍率・ゼミ・資格
  3. パスナビ等の進学情報サイト — 補完
- **進め方**: エージェント並列のWeb調査。1大学＝1調査単位とし、「現行データとの照合レポート」を作成 → 差分を確認して反映
- 1回のセッションで20〜30校ずつ、複数セッションに分けて全123校を消化
- 各値に `source_url` と `verified_date` を必ず記録。画面免責を「出典付き」に格上げする土台にする
- **取れなかった値は無理に埋めない** — `data_status: "unconfirmed"` として現行の「要確認」表示を継続

## 看板ゼミの方針（全学部×1件以上）

- 123校のWeb調査は1校ずつ公式サイトを回るので、同じ調査パスで「この学部の看板ゼミ・研究室を1つ拾う」を追加する（増分コスト小）
- 調査中に追加の手間なく複数見つかった場合は**最大3件まで**記載してよい（1件目=看板を先頭に）。ただし複数集めるために調査を深掘りはしない
- 形式: 名称＋テーマ1行＋出典URL＋検証日
- **特定できない学部は null のまま**（無理に埋めない）→ 表示は大学単位の `zemi_features` にフォールバック。カバー率6〜8割を現実的な目標とする
- ゼミは年度変動が激しいため、表示時は「年度により変更あり」の固定注記を付ける（フェーズ3のUI要件として引き継ぐ）

- 全650学部に dept_id・name・campus・hensachi・entry_methods・interest_tags・data_status が揃っていること（値が unconfirmed でもフィールドは存在）
- verified の値には source_url と verified_date が必ずあること（機械チェックスクリプトで検証）
- 現行650学部の偏差値と照合し、差分一覧レポートが残っていること
- 看板ゼミのカバー率を集計して報告すること（目標6〜8割、未達でも null 運用で可）
